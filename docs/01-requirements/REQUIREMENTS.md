# REQ 요구사항 정의서

> 생성일: 2026-04-01
> Gate 1 — PM 에이전트 작성

---

## 비전

- **배경**: 공공/기업 제안 업무에서 HWP 형식으로 작성된 제안요구사항(RFP)을 수작업으로 분석하여 상세요구사항으로 분해하는 과정이 반복적이고 시간 소모적이다. 이를 자동화하여 업무 효율을 높이기 위해 본 도구를 개발한다.
- **대상 사용자**: HWP 형식의 제안요구서를 분석하고 상세요구사항 정의서를 작성해야 하는 업무 담당자 (제안팀, 기획팀, PM 등)
- **핵심 가치**: HWP 파일 하나를 업로드하면, AI가 원본 요구사항을 계층적 상세요구사항으로 자동 분해하여 즉시 편집·다운로드 가능한 형태로 제공한다.
- **성공 기준**: HWP 파일 업로드 후 5분 이내에 상세요구사항 목록이 화면에 표시되고, 담당자가 웹 채팅으로 AI 결과를 수정한 뒤 엑셀 파일(2단계)을 다운로드할 수 있다.

---

## 기술 스택

- **프론트엔드**: React
  - 근거: 동적 테이블 행 증가, 채팅 UI, 파일 업로드 등 인터랙티브 요소가 많아 SPA 프레임워크 적합
- **백엔드**: Python / FastAPI
  - 근거: 기존 `hwp_ole_reader.py`, `hwp_body_parser.py`가 Python 기반이며 재활용 가능
- **AI**: Anthropic Claude SDK (`anthropic` Python 패키지)
- **데이터베이스**: 미정 — 세션 내 상태 관리만 필요할 경우 인메모리(서버 세션) 가능, 영속성 필요 시 SQLite 또는 PostgreSQL 검토
- **파일 파싱**: `olefile` 라이브러리 + 기존 커스텀 파서 재활용
- **배포**: 미정 (로컬 실행 우선, 추후 컨테이너화 고려)

> REQ-NNN: 그룹 | REQ-NNN-NN: 상세 요구사항
> AC 참조 규칙: `AC-NNN-NN` — 해당 REQ를 커버하는 AC ID / `없음: [이유]` — 의도적 미작성 시 이유 명시

### REQ-001: HWP 파일 업로드 및 파싱

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-001-01 | 사용자는 웹 화면에서 HWP 파일을 업로드할 수 있다 | Must | 구현완료 | AC-001-01 |
| REQ-001-02 | 업로드된 HWP 파일에서 요구사항 ID,  요구사항 분류,  요구사항 명칭, 요구사항 내용 4개 항목을 추출한다 | Must | 구현완료 | AC-001-02 |
| REQ-001-03 | HWP 파싱은 기존 `hwp_ole_reader.py`와 `hwp_body_parser.py`를 재활용한다 | Must | 구현완료 | AC-001-03 |
| REQ-001-04 | 파싱 실패 또는 지원하지 않는 파일 형식 업로드 시 오류 메시지를 화면에 표시한다 | Must | 구현완료 | AC-001-04 |

#### AC-001-01: HWP 파일 업로드

- **Given**: 사용자가 웹 브라우저에서 업로드 화면에 접속해 있다
- **When**: `.hwp` 확장자 파일을 파일 선택 다이얼로그 또는 드래그앤드롭으로 선택하여 업로드 버튼을 클릭한다
- **Then**: 파일이 서버에 전송되고, 업로드 진행 상태(로딩 인디케이터)가 표시된다

#### AC-001-02: 원본 요구사항 4개 항목 추출

- **Given**: 유효한 HWP 파일이 서버에 업로드되었다
- **When**: 서버가 파싱을 완료한다
- **Then**: 요구사항 ID, 분류, 명칭, 내용이 각각 구분된 데이터로 추출되어 프론트엔드에 반환된다

#### AC-001-03: 기존 파서 재활용

- **Given**: 백엔드 프로젝트에 `hwp_ole_reader.py`와 `hwp_body_parser.py`가 포함되어 있다
- **When**: HWP 파일 파싱 요청이 들어온다
- **Then**: 해당 두 파일의 클래스/함수를 import하여 파싱을 처리하며, 동일 기능을 중복 구현하지 않는다

#### AC-001-04: 파싱 오류 처리

- **Given**: 사용자가 HWP 파일이 아닌 파일(예: `.pdf`, `.docx`)을 업로드하거나, 손상된 HWP 파일을 업로드한다
- **When**: 서버가 파일 파싱을 시도한다
- **Then**: "지원하지 않는 파일 형식입니다" 또는 "파일을 파싱할 수 없습니다" 메시지가 화면에 표시되고, 사용자는 파일을 다시 선택할 수 있다

### REQ-002: AI 상세요구사항 자동 생성

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-002-01 | 파싱된 원본 요구사항을 Claude API에 전달하여 상세요구사항을 자동 생성한다 | Must | 구현완료 | AC-002-01 |
| REQ-002-02 | 상세요구사항은 원본 1건에 대해 N개의 하위 항목(REQ-NNN-01, REQ-NNN-02 ...)으로 생성된다 | Must | 구현완료 | AC-002-02 |
| REQ-002-03 | AI 생성 중 로딩 상태(진행률 또는 스피너)를 화면에 표시한다 | Should | 구현완료 | AC-002-03 |
| REQ-002-04 | Claude API 호출 실패 시 오류 메시지를 표시하고 재시도 버튼을 제공한다 | Must | 구현완료 | AC-002-04 |

#### AC-002-01: AI 상세요구사항 생성 요청

- **Given**: HWP 파싱이 완료되어 원본 요구사항 목록이 서버에 준비되어 있다
- **When**: 사용자가 "상세요구사항 생성" 버튼을 클릭한다
- **Then**: Claude API에 원본 요구사항 데이터가 전달되고, 응답으로 상세요구사항 목록이 반환된다

#### AC-002-02: 1:N 계층 구조 생성

- **Given**: Claude API가 상세요구사항 생성 응답을 반환했다
- **When**: 프론트엔드가 응답을 파싱한다
- **Then**: 각 원본 요구사항(예: REQ-001)에 대해 1개 이상의 상세요구사항(REQ-001-01, REQ-001-02 등)이 생성되며, 원본 ID와 상세 ID의 연결 관계가 유지된다

#### AC-002-03: AI 생성 로딩 표시

- **Given**: 사용자가 상세요구사항 생성을 요청했다
- **When**: Claude API 호출이 진행 중이다
- **Then**: 화면에 로딩 인디케이터(스피너 또는 진행률 바)가 표시되어 사용자가 처리 중임을 인지할 수 있다

#### AC-002-04: API 호출 실패 처리

- **Given**: Claude API 호출 중 네트워크 오류 또는 API 오류가 발생한다
- **When**: 서버가 오류 응답을 받는다
- **Then**: "AI 생성에 실패했습니다. 다시 시도해주세요." 메시지와 재시도 버튼이 표시된다

### REQ-003: 결과 화면 표시 (테이블 UI)

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-003-01 | 원본 요구사항 추출 결과를 테이블 형식으로 화면에 표시한다 | Must | 구현완료 | AC-003-01 |
| REQ-003-02 | AI가 생성한 상세요구사항을 테이블 형식으로 표시하며, 새 항목은 동적으로 행이 추가된다 | Must | 구현완료 | AC-003-02 |
| REQ-003-03 | 상세요구사항 테이블에서 각 행의 내용을 직접 수정(인라인 편집)할 수 있다 | Should | 구현완료 | AC-003-03 |
| REQ-003-04 | 화면은 정보량이 많아도 직관적으로 인식 가능하도록 설계한다 (컬럼 구분, 색상 구분 등) | Should | 구현완료 | AC-003-04 |

#### AC-003-01: 원본 요구사항 테이블 표시

- **Given**: HWP 파싱이 완료되었다
- **When**: 파싱 결과가 프론트엔드에 도착한다
- **Then**: 요구사항 ID, 분류, 명칭, 내용 4개 컬럼을 가진 테이블이 화면에 렌더링되고, 추출된 행 수만큼 테이블 행이 표시된다

#### AC-003-02: 상세요구사항 동적 테이블

- **Given**: AI 생성 결과가 도착하고 있다
- **When**: 각 상세요구사항 항목이 스트리밍 또는 일괄로 수신된다
- **Then**: 테이블에 행이 동적으로 추가되어 표시되며, 원본 요구사항 ID와 상세 요구사항 ID가 함께 표시된다

#### AC-003-04: UI 직관성

- **Given**: 원본 요구사항(10건 이상)과 상세요구사항이 동시에 화면에 표시되어 있다
- **When**: 처음 사용하는 사용자가 화면을 본다
- **Then**: 원본/상세 구분이 색상 또는 들여쓰기로 시각적으로 구분되고, 컬럼 헤더가 명확히 표시된다

#### AC-003-03: 인라인 편집

- **Given**: 상세요구사항 테이블이 화면에 표시되어 있다
- **When**: 사용자가 특정 셀을 클릭한다
- **Then**: 해당 셀이 편집 가능한 입력 필드로 전환되고, 내용을 수정 후 포커스를 벗어나면 변경 내용이 클라이언트 스토어에 반영된다. 서버와의 동기화는 REQ-008에서 처리한다.

### REQ-004: 채팅 기반 AI 수정

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-004-01 | 사용자는 웹 채팅 인터페이스를 통해 AI에게 상세요구사항 수정을 요청할 수 있다 | Must | 구현완료 | AC-004-01 |
| REQ-004-02 | 채팅에서 수정 요청이 처리되면 테이블의 해당 항목이 즉시 갱신된다 | Must | 구현완료 | AC-004-02 |
| REQ-004-03 | 채팅 대화 내역이 화면에 유지된다 (스크롤 가능) | Should | 구현완료 | AC-004-03 |

#### AC-004-01: 채팅으로 수정 요청

- **Given**: 상세요구사항 테이블이 화면에 표시되어 있다
- **When**: 사용자가 채팅 입력창에 수정 지시(예: "REQ-001-02의 내용을 더 구체적으로 작성해줘")를 입력하고 전송한다
- **Then**: Claude API에 현재 요구사항 컨텍스트와 함께 수정 요청이 전달되고, AI의 응답이 채팅창에 표시된다

#### AC-004-02: 채팅 수정 결과 테이블 반영

- **Given**: AI가 채팅을 통해 특정 요구사항에 대한 수정안을 응답했다
- **When**: 수정안이 프론트엔드에 도착한다
- **Then**: 테이블의 해당 행이 수정된 내용으로 즉시 업데이트되고, 변경된 행이 시각적으로 강조 표시된다

#### AC-004-03: 채팅 대화 내역 유지

- **Given**: 사용자가 여러 번의 채팅 대화를 진행했다
- **When**: 사용자가 채팅 영역을 스크롤한다
- **Then**: 이전 대화 내용(사용자 메시지 + AI 응답)이 시간 순서대로 모두 표시된다

### REQ-005: 엑셀 다운로드

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-005-01 | 1단계 다운로드: HWP에서 추출한 원본 요구사항만 포함한 엑셀 파일을 다운로드할 수 있다 | Must | 구현완료 | AC-005-01 |
| REQ-005-02 | 2단계 다운로드: 원본 요구사항과 AI가 생성(및 수정)한 상세요구사항을 모두 포함한 엑셀 파일을 다운로드할 수 있다 | Must | 구현완료 | AC-005-02 |
| REQ-005-03 | 다운로드되는 엑셀 파일의 컬럼 구조는 요구사항 ID, 분류, 명칭, 내용을 포함한다 | Must | 구현완료 | 없음: AC-005-01, AC-005-02에 포함 |

#### AC-005-01: 원본 추출 엑셀 다운로드 (1단계)

- **Given**: HWP 파싱이 완료되어 원본 요구사항이 화면에 표시되어 있다
- **When**: 사용자가 "1단계 다운로드" 버튼을 클릭한다
- **Then**: 요구사항 ID, 분류, 명칭, 내용 컬럼을 포함하는 `.xlsx` 파일이 브라우저에서 자동으로 다운로드된다

#### AC-005-02: 상세요구사항 포함 엑셀 다운로드 (2단계)

- **Given**: AI 상세요구사항 생성이 완료되고 사용자가 채팅으로 수정까지 완료했다
- **When**: 사용자가 "2단계 다운로드" 버튼을 클릭한다
- **Then**: 원본 요구사항과 각 원본에 대응하는 상세요구사항(REQ-NNN-NN 계층)이 모두 포함된 `.xlsx` 파일이 브라우저에서 자동으로 다운로드된다

### REQ-006: 시스템 아키텍처 및 비기능 요구사항

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-006-01 | 프론트엔드와 백엔드는 분리된 서비스로 구성한다 (API 통신 방식) | Must | 구현완료 | AC-006-01 |
| REQ-006-02 | HWP 파싱 로직은 기존 `../project-mgmt/backend/app/parser/`의 코드를 재활용한다 | Must | 구현완료 | AC-006-02 |
| REQ-006-03 | 파일 업로드는 서버에서 처리 후 임시 저장하며, 처리 완료 후 삭제한다 | Should | 구현완료 | AC-006-03 |
| REQ-006-04 | 단일 사용자 세션 내에서 업로드-파싱-AI생성-수정-다운로드 전 흐름이 끊김 없이 동작한다 | Must | 구현완료 | AC-006-04 |

#### AC-006-01: 프론트엔드/백엔드 분리

- **Given**: 시스템이 실행 중이다
- **When**: 프론트엔드(React)가 백엔드(FastAPI)에 요청을 보낸다
- **Then**: HTTP REST API를 통해 통신하며, 프론트엔드와 백엔드는 독립적으로 기동/종료 가능하다

#### AC-006-02: 파서 재활용

- **Given**: 백엔드 코드에 `../project-mgmt/backend/app/parser/`의 파서 파일이 복사 또는 참조되어 있다
- **When**: HWP 파싱이 실행된다
- **Then**: 신규 파싱 코드 없이 기존 `HWPOLEReader`, `HwpBodyParser` 클래스로 처리된다

#### AC-006-03: 임시 파일 삭제

- **Given**: HWP 파일이 서버에 업로드되어 임시 저장된다
- **When**: 파싱 처리가 완료(성공 또는 실패)된다
- **Then**: 서버의 임시 파일이 자동으로 삭제되고, 원본 파일이 서버에 잔류하지 않는다

#### AC-006-04: 단일 세션 연속 흐름

- **Given**: 사용자가 HWP 파일을 업로드했다
- **When**: 업로드-파싱-AI생성-수정-다운로드 단계를 순서대로 진행한다
- **Then**: 페이지 새로고침 없이 동일 화면에서 모든 단계가 완료되며, 각 단계 간 데이터가 유지된다

### REQ-007: AI 백엔드 선택 옵션

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-007-01 | 설정에서 AI 백엔드를 선택할 수 있다 (`anthropic_api` / `claude_code_sdk`) | Must | 구현완료 | AC-007-01 |
| REQ-007-02 | `claude_code_sdk` 선택 시 `claude-agent-sdk` 패키지의 `query()` API를 사용하여 상세요구사항을 생성한다 | Must | 구현완료 | AC-007-02 |
| REQ-007-03 | `claude_code_sdk` 선택 시 채팅 기능도 `claude-agent-sdk`의 `query()` API로 처리한다 | Must | 구현완료 | AC-007-03 |
| REQ-007-04 | 선택된 백엔드는 환경변수 또는 설정 파일로 관리한다 | Must | 구현완료 | AC-007-04 |

#### AC-007-01: AI 백엔드 설정 변경

- **Given**: 사용자가 설정 화면(또는 설정 파일)에 접근해 있다
- **When**: AI 백엔드를 `anthropic_api` 또는 `claude_code_sdk` 중 하나로 변경한다
- **Then**: 재시작 없이(또는 재시작 후) 해당 백엔드가 활성화되어 이후 AI 요청이 선택된 백엔드로 처리된다

#### AC-007-02: claude_code_sdk — 상세요구사항 생성

- **Given**: AI 백엔드가 `claude_code_sdk`로 설정되어 있다
- **When**: 사용자가 "상세요구사항 생성" 버튼을 클릭한다
- **Then**: 서버가 `claude-agent-sdk`의 `query()` API를 호출하여 상세요구사항을 생성하고, 결과가 화면에 정상 표시된다

#### AC-007-03: claude_code_sdk — 채팅 수정

- **Given**: AI 백엔드가 `claude_code_sdk`로 설정되어 있다
- **When**: 사용자가 채팅 입력창에 수정 지시를 입력하고 전송한다
- **Then**: 서버가 `claude-agent-sdk`의 `query()` API를 호출하여 수정 결과를 생성하고, 테이블이 정상 업데이트된다

#### AC-007-04: SSE 스트림 인터페이스 일관성

- **Given**: AI 백엔드가 `anthropic_api` 또는 `claude_code_sdk` 중 하나로 설정되어 있다
- **When**: 프론트엔드가 상세요구사항 생성 또는 채팅 수정을 요청한다
- **Then**: 두 백엔드 모두 동일한 SSE(Server-Sent Events) 스트림 형식으로 응답하며, 프론트엔드 코드 변경 없이 백엔드 전환이 가능하다

### REQ-008: 인라인 편집 서버 동기화

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-008-01 | 백엔드에 `PATCH /api/detail/{id}` 엔드포인트를 추가하여 특정 상세요구사항의 필드/값을 수정할 수 있다 | Must | 구현완료 | AC-008-01 |
| REQ-008-02 | 프론트엔드 `patchDetailReq` 액션은 인라인 편집 저장 시 `PATCH /api/detail/{id}` API를 호출하여 서버 state를 갱신한다 | Must | 구현완료 | AC-008-02 |
| REQ-008-03 | 인라인 편집 완료 후 서버와 프론트엔드(Zustand 스토어)의 상세요구사항 데이터가 항상 일치한다 | Must | 구현완료 | AC-008-03 |

#### AC-008-01: PATCH 엔드포인트 제공

- **Given**: 백엔드 서버가 실행 중이다
- **When**: 클라이언트가 `PATCH /api/detail/{id}` 요청을 `{ "field": "내용", "value": "수정값" }` 형식으로 전송한다
- **Then**: 서버가 해당 ID의 상세요구사항에서 지정한 field 값을 수정하고, 수정된 항목을 JSON으로 응답한다

#### AC-008-02: 인라인 편집 시 API 호출

- **Given**: 상세요구사항 테이블이 화면에 표시되어 있다
- **When**: 사용자가 셀을 인라인 편집하고 포커스를 벗어난다
- **Then**: 프론트엔드가 `PATCH /api/detail/{id}` API를 호출하고, 서버에서 성공 응답을 받은 뒤 Zustand 스토어를 갱신한다

#### AC-008-03: 서버-클라이언트 state 일관성

- **Given**: 사용자가 인라인 편집으로 상세요구사항을 수정했다
- **When**: 이후 채팅 수정 요청을 AI에 전송한다
- **Then**: AI가 수신하는 컨텍스트에 인라인 편집으로 변경된 최신 내용이 반영되어 있다

### REQ-009: claude_code_sdk 세션 기반 연속 실행

| ID | 요구사항 | 우선순위 | 상태 | AC 참조 |
|----|---------|---------|------|--------|
| REQ-009-01 | `claude_code_sdk` 백엔드 선택 시, 상세요구사항 최초 생성(`/api/generate`)을 세션 시작으로 삼아 반환된 `session_id`를 서버 state에 저장한다 | Must | 구현완료 | AC-009-01 |
| REQ-009-02 | 이후 채팅 요청(`/api/chat`)은 저장된 `session_id`를 `ClaudeAgentOptions(resume=session_id)`에 전달하여 동일 세션을 이어받아 처리한다 | Must | 구현완료 | AC-009-02 |
| REQ-009-03 | 새 HWP 파일 업로드 또는 "새로 생성" 버튼 클릭 시 서버 state의 `session_id`를 초기화한다 | Must | 구현완료 | AC-009-03 |

#### AC-009-01: 세션 ID 저장

- **Given**: AI 백엔드가 `claude_code_sdk`로 설정되어 있고, 사용자가 "상세요구사항 생성" 버튼을 클릭했다
- **When**: `claude-agent-sdk`의 `query()` 호출이 완료되어 상세요구사항이 생성된다
- **Then**: 해당 실행에서 반환된 `session_id`가 서버 state에 저장되고, 이후 채팅 요청에서 재사용 가능한 상태가 된다

#### AC-009-02: 세션 이어받기 채팅

- **Given**: `claude_code_sdk` 백엔드가 활성화되어 있고 서버 state에 유효한 `session_id`가 존재한다
- **When**: 사용자가 채팅 입력창에 수정 지시를 입력하고 전송한다
- **Then**: 서버가 `ClaudeAgentOptions(resume=session_id)`로 `query()`를 호출하여 이전 생성 세션을 이어받고, AI가 자신이 생성한 내용을 기억한 상태에서 수정 결과를 반환한다

#### AC-009-03: 세션 초기화

- **Given**: 서버 state에 기존 `session_id`가 저장되어 있다
- **When**: 사용자가 새 HWP 파일을 업로드하거나 "새로 생성" 버튼을 클릭한다
- **Then**: 서버 state의 `session_id`가 `None`으로 초기화되고, 다음 생성 요청은 새 세션으로 시작된다
